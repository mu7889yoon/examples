自作 FaaS ランタイム仕様書
Bun ランタイム採用
1. 目的

Bun をランタイムとして採用した自作 FaaS 実行基盤を定義する。本仕様は HTTP ベースの関数実行要求を受け付け、Docker コンテナ内で Bun ランタイムを使用して関数を実行し、その結果を返却するための仕様である。

2. ランタイム選定理由

Bun は JavaScript/TypeScript を高速に実行可能なランタイムであり、ランタイム単体で高速な起動と実行性能を有する点が評価された。Bun は JavaScriptCore エンジン（WebKit ベース）を利用し、Zig 言語で実装されたオールインワンランタイムである。これにより Node.js と比較して起動時間が短い等の利点がある。また Bun はパッケージ管理やバンドラ、テストランナーなど複数機能を内蔵する設計となっているが、実行時はランタイム機能のみを使用する。

3. 実行モデル
3.1 処理要求受け渡し

FaaS 実行基盤は具体的に次の流れで関数を実行する。

クライアントは HTTP API で関数実行要求を送信する。

FaaS サーバは対象関数の Docker イメージを確認する。

対象関数は Docker コンテナとして起動し、Bun ランタイムで実行される。

標準出力から出力を取得しクライアントに JSON 応答する。

3.2 Docker 実行戦略

関数はコンテナにより隔離実行される。実行方法には次の 2 種類を定義する。

3.2.a Cold Start 実行

関数呼び出しごとにコンテナを起動し、Bun で関数コードを実行する。

例:

docker run --rm -i --name faas_<name> \
  -v <path_to_function_code>:/app \
  -w /app bun:latest bun run handler.js


この実行は関数ごとに新規コンテナが生成される Cold Start 方式である。

3.2.b Warm Running 実行（オプション）

起動済みコンテナを保持し、同一コンテナ内で複数リクエストを受け付ける方式とする。Warm Running 実行ではコンテナ内で HTTP サーバーを起動し、Bun の serve() 等によるリクエスト処理を行う。

4. 関数コード要件

関数は Bun で実行可能な JavaScript/TypeScript で記述する。関数ハンドラは次の形式を持つこと。

4.1 handler.js 例
export default async function handler(event) {
  return {
    statusCode: 200,
    body: `Hello ${event.body}`
  };
}


event オブジェクトには JSON ボディとメタ情報を含む。

5. エンドポイント仕様
5.1 関数実行
POST /function/<name>


要求 JSON 形式

{
  "body": {...},
  "metadata": {...}
}


応答 JSON 形式

{
  "statusCode": 200,
  "body": "<string>"
}

5.2 関数登録
POST /register


要求 JSON

{
  "name": "<function-name>",
  "image": "<docker-image>"
}


関数登録では functions.json 等の登録リストに追加し、自動でイメージ Pull を行う。

6. ランタイム挙動
6.1 起動時間最適化

Bun ランタイムの利用により、プロセス起動時間を短縮することを目指す。Bun は起動時間が比較的短く設計されているため、Cold Start 時のレイテンシ低減に寄与する可能性がある。

6.2 言語互換性

Bun は Node.js の API との互換性を目指しているが、すべての Node.js API が完全対応しているわけではない。そのため関数内で利用するモジュールや API に関しては互換性を検証済みであることが望ましい。

7. エラー処理

関数実行中にエラーが発生した場合、HTTP ステータス 500 を返し、body にエラーメッセージを含める。

例

{
  "statusCode": 500,
  "body": "Error: <message>"
}

8. セキュリティ要件

ランタイムコンテナおよび FaaS サーバは root 権限で実行しないことを推奨する。

Docker コマンド実行時の引数はホワイトリストまたは検証ルールに従いサニタイズする。

外部から直接 Docker API を叩かれないよう適切なアクセス制御を実装する。

9. ロギングおよび監視

実行ログはサーバ側で標準出力および標準エラーを収集し、時間・関数名・ステータスをログとして保持する。

10. 制約および注意点

Bun は高速なランタイムであるが、Node.js 互換 API の完全性は保証されていないため、依存する npm モジュールの動作検証が必要である。

メモリ消費や I/O 性能はランタイム選択だけでなく Docker コンテナ設定に依存する。

Warm Running 実行方式はリソース消費が増えるため、Raspberry Pi のような低リソース環境では慎重に設計する。

11. 付録：ランタイム環境構成

Docker イメージは Bun の公式またはバンドル済みイメージを使用する。

コンテナは関数名に基づく命名規則を採用する（例: faas_<name>）。

ランタイム設定は環境変数に依存せず、明示的な設定ファイルで管理する。
